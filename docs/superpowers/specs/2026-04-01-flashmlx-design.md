# FlashMLX Design Spec

An auto-research driven MLX inference engine for Apple Silicon. Build a minimal, modular inference engine, then point an autonomous research loop at it to iteratively discover faster configurations across model architectures.

## Goals

- Maximize tokens/second on Apple Silicon for diverse model architectures (Llama-family, MoE, Gemma)
- Use an autoresearch-inspired loop to autonomously explore the optimization space
- Discover Pareto frontier configurations across architectures, speed, memory, and quality
- Keep perplexity within 1% of FP16 baseline (correctness hard gate)

## Non-Goals

- No HTTP server or API layer (kernel-level engine only)
- No multi-model serving or continuous batching (single model, single stream)
- No custom Metal shaders in v1 (MLX built-in kernels + lazy evaluation only)
- No support for non-Apple-Silicon hardware

---

## System Architecture

Two cleanly separated components:

### FlashMLX Engine (mutable — autoresearch optimizes this)

The inference engine is a pipeline of composable modules with clean interfaces.

### AutoBench (immutable — the optimization loop)

An autoresearch-style loop that proposes changes, benchmarks them, and keeps improvements.

```
flashmlx/
├── engine/                    # MUTABLE — autoresearch modifies these
│   ├── attention.py           # Attention kernels (standard, flash, GQA, MQA, MoE routing)
│   ├── kv_cache.py            # KV cache (naive, paged, prefix-cached, quantized)
│   ├── quantize.py            # Weight quantization (4/6/8-bit, mixed-precision)
│   ├── generate.py            # Decode loop (autoregressive, speculative, chunked prefill)
│   └── model_config.py        # Architecture dispatch & model loading
│
├── harness/                   # IMMUTABLE — autoresearch cannot touch these
│   ├── prepare.py             # Downloads models, generates reference outputs, caches eval data
│   ├── benchmark.py           # Runs engine, measures tok/s, TTFT, memory, perplexity
│   ├── validate.py            # Correctness gate: perplexity < 1% degradation from baseline
│   ├── models.yaml            # Model registry (name, HF repo, architecture type, quant format)
│   └── baselines.json         # Generated by prepare.py — baseline metrics per model per hardware
│
├── autobench/                 # IMMUTABLE — the autoresearch loop itself
│   ├── program.md             # Natural language research directives (mutable by HUMAN only)
│   ├── loop.py                # Core loop: propose → commit → benchmark → keep/discard
│   ├── results.tsv            # Experiment log (never committed to git)
│   └── analyze.py             # Post-hoc analysis: Pareto plots, regression detection
│
├── baselines/                 # Reference benchmarks from other engines
│   └── compare.py             # Run same prompts through mlx-lm / ollama for comparison
│
└── docs/
    └── superpowers/specs/     # Design docs
```

### Key Invariants

- `harness/` and `autobench/` are never modified by the autoresearch agent
- `engine/` is the sole optimization surface
- `program.md` is the human's control interface for steering research direction
- `results.tsv` logs every experiment (keeps, discards, crashes) but is never committed to git
- Git branch always points to the best-performing configuration
- No new packages can be installed — engine uses only `mlx`, `mlx-nn`, and standard library

## FlashMLX Engine — Module Design

### Model Loading & Architecture Dispatch (`model_config.py`)

```python
load_model(path: str, quant: QuantConfig) -> Model
```

Reads `architectures` field from HF `config.json` and maps to the right attention/FFN pattern.

Supported architecture families:
- **LlamaFamily**: Llama 3, Qwen, Mistral (GQA decoder-only)
- **MoEFamily**: Mixtral, DeepSeek (expert routing + shared experts)
- **GemmaFamily**: Gemma 2/3 (sliding window + global attention interleaving)

Autoresearch can modify how layers are composed, how weights are loaded, and how architecture-specific logic is dispatched. The entire file is mutable.

### Attention (`attention.py`)

```python
attend(q, k, v, mask, cache: KVCache) -> output
```

Initial implementations:
- `standard_attention` — naive scaled dot-product (baseline)
- `flash_attention` — tiled, memory-efficient (MLX metal kernel)
- `grouped_query_attention` — GQA with configurable num_kv_heads

Note: MoE expert routing is part of the FFN layer, not attention. `model_config.py` handles MoE-specific FFN dispatch (top-k gating, expert selection, shared experts).

Optimization surface for autoresearch: tile sizes, head grouping strategies, fused operations, attention approximations, lazy evaluation grouping to maximize Metal kernel fusion.

### KV Cache (`kv_cache.py`)

```python
class KVCache:
    def update(layer_idx, keys, values) -> None
    def get(layer_idx) -> (keys, values)
    def prefix_match(token_ids) -> cache_hit_length
```

Initial implementations:
- `NaiveCache` — simple list of tensors per layer
- `QuantizedCache` — stores K/V in reduced precision (q8/q4)
- `PagedCache` — fixed-size blocks with block table lookup
- `PrefixCache` — hash-based prefix matching for prompt reuse

Optimization surface: block sizes, eviction policies, quantization formats, hybrid strategies (paged + prefix), per-layer cache precision.

### Quantization (`quantize.py`)

```python
quantize_weights(model, config: QuantConfig) -> Model
```

`QuantConfig` specifies per-layer or global:
- `bit_width`: 3, 4, 6, 8, 16
- `group_size`: 32, 64, 128
- `scheme`: symmetric, asymmetric, mixed-bit (sensitivity-based)

Optimization surface: optimal mixed-precision configurations — which layers tolerate aggressive quantization, which need higher precision.

### Generation Loop (`generate.py`)

```python
generate(model, prompt_tokens, max_tokens, ...) -> Iterator[token]
```

Strategies:
- `autoregressive` — standard one-token-at-a-time
- `speculative` — draft model proposes N tokens, main model verifies
- `chunked_prefill` — process prompt in chunks to avoid GPU stalls

Optimization surface: prefill chunk sizes, speculative draft length, sampling strategies, early stopping heuristics.

### Data Flow

```
prompt_tokens
    → model_config.load_model(path, quant_config)
    → quantize.quantize_weights(model, quant_config)
    → generate.chunked_prefill(model, prompt_tokens, kv_cache)
    → loop:
        → attention.attend(q, k, v, mask, kv_cache)
        → kv_cache.update(layer_idx, new_k, new_v)
        → sample next token
        → yield token
```

## AutoBench — The Optimization Loop

### Core Loop (`loop.py`)

Each experiment follows this sequence:

1. Read current state (`git log`, `results.tsv`)
2. Select target module based on current cycle
3. Propose a modification (agent reads module, analyzes past results)
4. Commit the change
5. Run benchmark suite (small model: ~30-60s)
6. Extract metrics: tok/s, TTFT, memory_mb, perplexity
7. Correctness gate: if perplexity > baseline * 1.01 → REJECT
8. Composite score: `0.7 * normalized_tok_s + 0.3 * normalized_ttft`
9. Keep if score > best_score, else `git reset --hard HEAD~1`
10. Every Kth experiment: validate on medium model (confirm generalization)
11. Log to `results.tsv` regardless of outcome
12. Repeat

### Cycling Strategy

`program.md` directs exploration in structured rounds:

```
Round 1 (experiments 1-20):   attention.py     — kernel optimizations, tiling, fusion
Round 2 (experiments 21-40):  kv_cache.py      — cache strategies, quantization, paging
Round 3 (experiments 41-60):  quantize.py      — mixed-precision, group sizes, per-layer config
Round 4 (experiments 61-80):  generate.py      — decode loop, speculative decoding, prefill
Round 5 (experiments 81-100): cross-module      — co-optimize across modules freely
```

After round 5, the cycle restarts with a higher baseline. Each cycle builds on the previous best.

### `program.md` — Human Control Interface

The only file the human edits to steer research. Contains:
- Objective statement (what to optimize)
- Current cycle directives (which module, what to try)
- Hard constraints (immutable files, no new packages, simplicity preference)
- Hardware context (auto-detected at startup)
- `NEVER STOP` directive for autonomous operation

### `results.tsv` Format

```
commit | module | tok_s | ttft_ms | memory_mb | perplexity | ppl_delta_pct | composite_score | status | description
a3f9c2 | attention | 142.3 | 45.2 | 4820 | 8.12 | 0.2% | 0.83 | keep | fused qkv+attn metal kernel
b7e1d4 | attention | 138.1 | 51.0 | 4815 | 8.15 | 0.6% | 0.79 | discard | tiled attn tile_size=256
c2a8f1 | attention | 0 | 0 | 0 | 0 | 0 | 0 | crash | rope freq scaling overflow
```

### Validation Tier

Every 10th experiment, the loop runs the current configuration against the medium model suite (3-8B). If the medium model score drops below 95% of its baseline, the result is flagged for human review (not auto-discarded — the optimization might still be valuable for small models).

### Crash Handling

If an experiment crashes:
1. Read the last 50 lines of the run log for the stack trace
2. Attempt a simple fix (typo, import, shape mismatch) and retry once
3. If the retry also crashes, log as `status=crash` and `git reset --hard HEAD~1`
4. Move on to the next experiment

## Benchmark Harness

### Model Registry (`models.yaml`)

**Small tier (fast iteration):**
- `qwen3-0.6b` — mlx-community/Qwen3-0.6B-4bit (Llama-family, primary iteration target)
- `smollm2-1.7b` — mlx-community/SmolLM2-1.7B-4bit (Llama-family, secondary)

**Medium tier (validation):**
- `llama3-8b` — mlx-community/Meta-Llama-3-8B-4bit (Llama-family)
- `mistral-7b` — mlx-community/Mistral-7B-Instruct-v0.3-4bit (Llama-family)
- `deepseek-moe` — mlx-community/deepseek-moe-16b-base-4bit (MoE-family, tests expert routing)

### Benchmark Protocol (`benchmark.py`)

Each benchmark run:

1. Load model with current engine configuration
2. Warm-up: generate 20 tokens (discard — primes Metal shader caches)
3. Prefill benchmark: short prompt (128 tokens) and long prompt (1024 tokens) × 3 runs each → median TTFT
4. Generation benchmark: generate 256 tokens from a fixed prompt × 3 runs → median tok/s
5. Memory snapshot: peak Metal memory during generation
6. Return: `{tok_s, ttft_short_ms, ttft_long_ms, memory_mb}`

Three runs with median eliminates noise from Metal shader compilation and system load variance.

### Correctness Validation (`validate.py`)

Runs separately from speed benchmarks:

1. Load FP16 baseline perplexity from `baselines.json` (computed once by `prepare.py`)
2. Run current engine on eval dataset (WikiText-2 subset, ~500 sequences)
3. Compute perplexity on generated logits
4. `ppl_delta_pct = (current_ppl - baseline_ppl) / baseline_ppl * 100`
5. PASS if `ppl_delta_pct < 1.0%`, FAIL otherwise

### `prepare.py` — One-Time Setup

Run once before the first autobench cycle:

1. Download all models from `models.yaml` via `huggingface_hub`
2. Generate baseline metrics for each model (FP16 perplexity, baseline tok/s, TTFT, memory with naive engine)
3. Cache eval dataset (tokenized WikiText-2 subset)
4. Save baselines to `harness/baselines.json` with hardware context (chip, cores, bandwidth via `system_profiler`)
5. Verify engine can load and generate for every model in registry

### Timing Budget

**Per experiment (small model):** ~60s total
- Warm-up: ~5s
- Prefill bench: ~10s
- Generation bench: ~15s
- Validation: ~20s
- Overhead (git, logging): ~10s

**Validation (medium model, every 10th):** ~10 min (3-4 min per model × 2-3 models)

**Overnight (8 hours):** ~300-400 experiments with periodic medium-model validation

## Technology Choices

**Runtime:**
- Python 3.11+
- `mlx` and `mlx-nn` — core framework, Metal acceleration
- `huggingface_hub` — model downloading
- `transformers` — tokenizers only
- `numpy` — metric computation in validation

**Autobench agent:**
- Claude (or any LLM with tool use) invoked via API
- Agent receives: `program.md`, current module source, `results.tsv` tail, git log
- Agent returns: a diff to apply to the target module
- Simple prompt → diff → apply → benchmark cycle (no agentic framework)
- API key via environment variable

**Hard constraints:**
- No additional Python packages — engine uses only `mlx`, `mlx-nn`, and standard library
- No custom Metal shaders in v1 — rely on MLX's lazy evaluation and kernel fusion
- Git is the sole checkpoint system (no MLflow, W&B, etc.)

**Minimum hardware:** Any Apple Silicon Mac. Optimizations are hardware-specific by design. `prepare.py` records chip/cores/bandwidth in `baselines.json` for contextualization.

## Post-Hoc Analysis (`analyze.py`)

After an overnight run, generates:
- **Pareto frontier plot**: tok/s vs. memory vs. perplexity across all kept configurations
- **Best configuration per architecture family**: which engine settings work best for Llama vs. MoE
- **Regression detection**: which changes hurt which model sizes
- **Cumulative improvement curve**: tok/s over experiment number, showing rate of discovery
- **Module attribution**: which module's experiments yielded the most improvement

## Future Extensions (Not In Scope for v1)

- Custom Metal kernels as an additional optimization surface
- Population-based search (multiple parallel branches exploring different strategies)
- HTTP server layer with continuous batching (built on top of optimized engine)
- Multi-objective Pareto keep/discard (replace composite score with dominance check)
- Automatic `program.md` evolution (agent suggests next round's directives based on results)
